5) Занимают ли зомби-процессы какие-то ресурсы в ОС (CPU, RAM, IO)?

Ответ:

Создаем файл со python скриптом ( https://ru.wikipedia.org/wiki/wiki/Процесс-зомби )

         // #!/usr/bin/env python2
         // # -*- coding: utf8 -*-
         
         import subprocess
         import time
         import threading
         
         // # Порождаем процесс 'ls' с параметром '-l'
         proc = subprocess.Popen(['ls','-l'])
         
         // # Останавливаем главный поток программы на 5 секунд. На их протяжении процесс будет иметь
         // # статус "зомби", хоть процесс уже и завершился, поскольку не была выполнена обработка
         // #результата процесса
         time.sleep(5)
         
         // # В этом месте зомби исчезнет, т.к. программа очистит буферы ввода-вывода для процесса
         // # и считает код его завершения
         proc.communicate()
         
         time.sleep(5)

7) Чем отличается последовательность команд через ; и через && в bash? Например:
   
Ответ:

   Оболочка BASH просто выполнит эти команды последовательно:  

       root@netology1:~# test -d /tmp/some_dir; echo Hi
       Hi

   Оболочка BASH выполнит вторую команду поcле оператора AND (&&)
   только в том случае, если при выполнении первой команды SUCCEEDS, 
   т.е. состояние выхода первой команды равно «0»  (Успех).

   Ниже команда test проверила, что  файл /tmp/some_dir не существует 
   и не является каталогом и получила в выходной поток stdout значение ошибки - "1" .
   Поэтому последовательность команд был остановлена.

     root@netology1:~# test -d /tmp/some_dir && echo Hi   
     root@netology1:~#   

Есть ли смысл использовать в bash &&, если применить set -e?

Ответ:

    Это не имеет смысла, так как вторая команда никогда не будет выполнена.
    Если первая команда неуспешна (В выходной потоке stdout - "1") - работа конвейера будет завершена принудительно из-за set -e.
    Если первая команда успешна (В выходном потоке stdout -"0" ) - работа конвейера по правилам оператора "&&" прекратится.

8)  Из каких опций состоит режим bash set -euxo pipefail и почему его хорошо было бы использовать в сценариях?

Опции режима bash set -euxo pipefail

    -e Прерывает процесс, если результат выполнения очередной команды не равен "0"(Успех) .
       Заставляет скрипт завершить работу, если любой из вызываемых им процессов 
       выдает ненулевой код возврата. Все, что не равно нулю, считается неудачей.

    -x Печатать команды и их аргументы по мере их выполнения.

    -o pipefail.  
       Статусом выхода из конвейера, в том случае, если не включена опция pipefail, 
       служит статус завершения последней команды конвейера. 
       Если опция pipefail включена — Статус выхода из конвейера равен статусу выхода последней (самой правой) 
       команды, завершённой с ненулевым статусом ( "Ошибка"), или "0" — если работа всех команд завершена успешно.

    -u Рассматривать неинициализированные переменные как ошибку при подстановке -1 ( "Ошибка").
       Вывести в stderr текст ошибки и завершить неинтерактивный вызов.
       Неинициализированные переменные очень трудно обнаружить в реальном сценарии.
       При этом сообщение об ошибке не появляется. Остальная часть сценария продолжит выполняться.

  В сценариях удобно использовать вышеуказанные опции, потому что команды выводятся на экран одна за другой, 
  и последней выведенной команда и аргументами будет та команда, которая выполнена со статусом ошибки.
  Также можно легко посмотреть ее статус ошибки.

9) Используя -o stat для ps, определите, какой наиболее часто встречающийся статус у процессов в системе. 
   В man ps ознакомьтесь (/PROCESS STATE CODES) что значат дополнительные к основной заглавной буквы статуса процессов. 
   Его можно не учитывать при расчете (считать S, Ss или Ssl равнозначными).

Ответ :

   Наиболее частый статус процессов это:
  
        S - В  состоянии ожидания  "interruptible sleep" (waiting for an event to complete)

        I - Бездействующие потоки ядра.

   Дополнительные аттрибуты статуса S

        < Высокий приоритет                      high-priority (not nice to other users)

        N Низкий приоритет                       low-priority (nice to other users)

        L Имеет страницы, залоченные в памяти    has pages locked into memory (for real-time and custom IO)

        s Основатель сессии                      is a session leader

        l Мультипотоковый                        is multi-threaded (using CLONE_THREAD, like NPTL pthreads do)

        + Находится в группе процессов переднего плана   is in the foreground process group
